## Proposed Changes (Configuration‑Only)

Below each component lists the **files or artefacts** that will be created/updated **only for configuration** – they do not modify kernel logic.

---

### 1️⃣ Observability & Monitoring Layer

#### Files / Artefacts
- **`ops/config/posthog.env`** – read‑only API key & project ID.
- **`ops/config/grafana.yaml`** – datasource definitions (Prometheus proxy, PostHog JSON endpoint) and dashboard import JSON (already generated in `ops/observability/grafana_dashboard.json`).
- **`ops/config/otel_collector.yaml`** – exporter set to *file* (`/tmp/otel_spans.json`) with *no‑export* to remote back‑ends.
- **`ops/scripts/health_sync.sh`** – already created; point to the new env vars.
- **`ops/scripts/trace_check.js`** – already created; ensure `OTEL_SPANS_FILE` points to `/tmp/otel_spans.json`.
- **`ops/scripts/poe_validator.sh`** – already created; reference `posthog.env`.

#### Actions
1. Populate the env files with the production secrets (read‑only keys).
2. Add the Grafana datasource config to the Grafana provisioning directory (`/etc/grafana/provisioning/datasources/`).
3. Load the dashboard JSON via Grafana API or provisioning (`dashboard.yaml`).
4. Set up a **Prometheus** instance that scrapes the health‑sync metric endpoint (`/metrics` exposed by `health_sync.sh` if you enable the optional `--export-metrics` flag).
5. Wire alerts in **Alertmanager**: 
   - `metric_drift > 1%`
   - `poe_hash_mismatch`
   - `trace_chain_missing`
   - Route to a Slack/Email channel.

### 2️⃣ Secure Secret Management

#### Files / Artefacts
- **`ops/secrets/vault_policy.hcl`** – read‑only policy for PostHog & Grafana secrets.
- **`ops/secrets/vault_init.sh`** – one‑time script that writes the secrets to HashiCorp Vault (or AWS Secrets Manager) and locks the path.

#### Actions
1. Initialise Vault (or your chosen secret store) in **read‑only** mode.
2. Store `POSTHOG_READ_ONLY_API_KEY`, `POSTHOG_PROJECT_ID`, and `GRAFANA_API_KEY`.
3. Grant the SIMIS service‑account **read** capabilities only.
4. Verify that the env files (`posthog.env`) are populated at container start via `vault read` (no write permissions).

---

### 3️⃣ Compliance & Certification Artefacts

#### Files / Artefacts
- **`docs/observability_status_report.md`** – generated by `ops/scripts/generate_report.sh` (snapshot of logs).
- **`docs/compliance_matrix.md`** – mapping of ISO 27001 / NIST 800‑53 controls to SIMIS components.
- **`docs/SIMIS‑PCDES‑CERT‑v1.0.pdf`** – final certificate (signed by internal CA).
- **`ops/artifacts/immutable_snapshot.sha256`** – SHA‑256 of the exported IOBuffer ledger (uploaded to WORM storage).

#### Actions
1. Run `./ops/scripts/generate_report.sh` after a stable hour of operation.
2. Export the IOBuffer ledger (`ops/scripts/export_iobuffer.sh` – read‑only dump to JSON).  
3. Compute the hash and store it in `immutable_snapshot.sha256`.
4. Upload `observability_status_report.md`, `compliance_matrix.md`, `SIMIS‑PCDES‑CERT‑v1.0.pdf`, and the snapshot bundle to the **immutable storage** bucket.
5. Request the external auditor to sign‑off on the uploaded artefacts.

---

### 4️⃣ Production Deployment Pipeline (CI/CD)

#### Files / Artefacts
- **`.github/workflows/observability_deploy.yml`** – GitHub Actions workflow that:
  1. Lints configuration files (YAML/JSON).
  2. Deploys Grafana dashboards via its API.
  3. Pushes env files to the secret store (read‑only).
  4. Runs `generate_report.sh` and uploads artefacts to immutable bucket.
- **`ops/ci/deploy.sh`** – wrapper script used by the workflow to start the three daemons inside a Docker compose stack (`docker-compose.yml` already contains only observability services).

#### Actions
1. Enable the workflow on the `main` branch; it runs on every merge to `release‑candidate`.
2. The workflow **fails fast** if any validation step (health sync drift, PoE hash mismatch, trace chain) returns a non‑zero exit code.
3. On successful run, a Git tag `simis‑prod‑<YYYYMMDD>` is created.
4. The tag triggers a downstream deployment to the production Kubernetes namespace (only the observability pods, no kernel pod).

---

### 5️⃣ Operational Runbooks & SLA

#### Files / Artefacts

- **`docs/runbook/maintenance_window.md`** – procedure for rotating API keys, renewing certificates, and performing the yearly immutable snapshot export.
- **`docs/sla.md`** – defined Service Level Agreement (99.9 % dashboard availability, < 5 min mean‑time‑to‑acknowledge alerts).

#### Actions
1. Publish the runbooks to the internal Confluence/Wiki.
2. Conduct a tabletop exercise with the ops team to validate the incident flow.
3. Record the SLA metrics in Grafana and expose them via the `observability_status_report.md`.
